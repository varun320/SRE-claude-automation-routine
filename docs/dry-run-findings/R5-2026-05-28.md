## CRITICAL FINDING 1 — D7 is wrong; the "10 shared mailboxes" are mostly aliases

**Evidence:** I called `list-mail-folders` on `info-me@sulfurrecovery.com` and `boardroom@sulfurrecovery.com`. Both returned **the exact same folder structure as Maaz's primary mailbox**:

- Same folder IDs (Archive, Conversation History, Deleted Items, Drafts, emails)
- Same Deleted Items unread count: 151
- Same Deleted Items total: 913
- Same Drafts count: 2
- Same byte sizes

This is Graph's signal that those addresses resolve to Maaz's primary mailbox — i.e., they're **email aliases**, not distinct mailboxes.

**Implications for D7:**

| Address | Reality |
|---|---|
| `info@sulfurrecovery.com` | Real separate mailbox (own item store) |
| `careers@sulfurrecovery.com` | Real separate mailbox (low volume, 0 in 24h window) |
| `info-me@sulfurrecovery.com` | **Alias** on Maaz's primary |
| `boardroom@sulfurrecovery.com` | **Alias** on Maaz's primary |
| `ar@sulfurrecovery.com` | Presumed alias (same 404 pattern as confirmed aliases) |
| `ap@sulfurrecovery.com` | Presumed alias |
| `sales@sulfurrecovery.com` | Presumed alias |
| `apple@sulfurrecovery.com` | Presumed alias |
| `HusamsBookingPageSRE@sulfurrecovery.com` | Presumed alias |
| scanner (disabled) | Out of scope already |

**Operational meaning:** Mail addressed to any alias lands directly in Maaz's primary inbox with `toRecipients` containing the alias address. So R5 does NOT need to query 7 separate mailboxes — those addresses are caught by querying Maaz's primary once and matching `toRecipients`.

**Required spec fixes:**

1. **D6/D7 filter expansion** — R1 and R5 both need to expand the `toRecipients` keep-list from `{maaz@, info@, info-me@}` to:
   ```
   {maaz@, info@, info-me@, ar@, ap@, sales@, apple@, boardroom@, HusamsBookingPageSRE@}
   ```
   Without this, mail addressed to ar@/ap@/sales@/apple@/boardroom@/HusamsBookingPageSRE@ gets silently dropped as cross-mailbox pollution. Today's window happened to have no such items so nothing surfaced as a bug — but this is a latent dropout risk.

2. **R5 build prompt** — replace the per-mailbox enumeration over 9 mailboxes with two queries: Maaz primary (catches all alias-routed mail) + `info@` (separate stream) + `careers@` (separate stream, low volume). The 7 aliases get caught via Maaz primary's `toRecipients` filter.

3. **client-config.yaml.draft** — restructure `shared_mailboxes` into two groups: `independent_mailboxes` (info@, careers@) and `aliases_routing_to_primary` (info-me@, boardroom@, plus the 5 presumed-alias addresses pending verification).

4. **Verification step** — confirm the 5 presumed-aliases (ar@, ap@, sales@, apple@, HusamsBookingPageSRE@) actually do route to Maaz's primary (vs being disabled accounts that bounce). Quick test: send a test message to each and confirm it lands in Maaz's inbox.

## CRITICAL FINDING 2 — R1's narrow D6 filter is silently dropping alias mail

Same root cause as Finding 1. R1's `toRecipients` keep-list is `{maaz@, info@, info-me@}` — anything addressed to the other 6 alias addresses gets dropped as "cross-mailbox pollution" even though it landed legitimately in Maaz's primary.

R1's 2026-05-28 dry-run window happened to have zero such items, so the issue didn't surface as a bug — but this is a latent dropout. The fix is the same expansion as Finding 1.

## CRITICAL FINDING 3 — D6 needs deduplication when querying multiple mailboxes

Anthony Grossu's "Any loads coming up?" appeared in BOTH Maaz primary AND info@'s 24h query — same sender, same subject, same timestamp. Likely the message landed in info@'s real mailbox AND was auto-forwarded / had a transport rule copy it to Maaz primary.

Without dedup, R5 surfaces the same item twice. Today's window had ~4 such overlaps.

**Recommended fix:** add a dedup step keyed on `internetMessageId` (Graph property) or fallback `(senderAddress, subject, receivedDateTime within ±60s)` after merging Maaz primary + info@ + careers@.
