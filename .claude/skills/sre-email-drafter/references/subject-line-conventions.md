# Subject-line conventions

Observed from the tenant audit's sent-items sample. These patterns are how Maaz's emails get triaged on the recipient side — getting them right means the email reaches the right priority queue.

---

## Pattern 1 — DRAFT for AIMS approval before client send

When Maaz sends a draft TO AIMS for review BEFORE it goes to the end-customer:

```
DRAFT for your approval - <Customer Name> (<Customer Contact>): <Topic>
```

**Real example from audit**:
> `DRAFT for your approval - QChem (Sanjay Bhatt): SGRU/SRU 2028 Feed Study and SRU/AGRU Training`

Recipient is the AIMS bridge contact (e.g., `bader@aimsgt.com`). The end-customer is NOT cc'd at this stage. AIMS reviews, approves or edits, and then either Maaz or AIMS sends to the end-customer.

---

## Pattern 2 — Forward to internal team after external send

When Maaz forwards an external email/PO to the internal SRE team for awareness or action:

```
FW: <original subject>
```

**Real example**:
> `FW: SRU-1 Onsite Testing 6511226695 Approved *PO Signing*`

Recipients are typically Dharmesh, Ashley, Talha (+ project-specific roles). The asterisk-wrapped tag `*PO Signing*` is a Maaz convention to flag closing events in subject lines.

---

## Pattern 3 — Reply chain (RE:)

Standard reply patterns:

```
Re: <original subject>
```

Or for explicit quoting:

```
RE: <original subject>
```

Both are fine. Maaz uses `Re:` for thread continuation, `RE:` when prefacing a deliberate new turn in the thread.

---

## Pattern 4 — AR / Finance threads

```
Re: AUDIT CONFIRMATION REQUEST: BALANCE AS OF 31.12.<YYYY> (SRE)
```

```
Re: Balance confirmation - SRE
```

```
FW: (Reminder-<N>-) <Customer> Online - Account Reconciliation <MM/YYYY>
```

**Real examples**:
- `Re: AUDIT CONFIRMATION REQUEST: BALANCE AS OF 31.12.2025 (SRE)` (year-end audit reply)
- `FW: (Reminder-5-) Tüpraş Online - Account Reconciliation 03/2026` (escalation #5)
- `Re: Balance confirmation- SRE`

The `(Reminder-N-)` count is significant — escalation #5 means it's been chased four times already. This is a flag for ar-followup template.

## Pattern 5 — Meeting threads

Outlook auto-generates these when accepting/declining/proposing new times:

```
Accepted: <meeting subject>
Tentative: <meeting subject>
Declined: <meeting subject>
New Time Proposed: <meeting subject>
```

Examples:
- `Accepted: Envent/ Sulfur Recovery Meeting at Envent`
- `Tentative: RTR SRUs Performance and Assessment Reports`
- `Tentative: Bi-Weekly Job Meeting and Planning`
- `Tentative: Maaz-Ron meeting on Teams`
- `New Time Proposed: Maaz-Ron meeting on Teams`

These are auto-generated by Outlook — don't draft them via this skill.

---

## Pattern 6 — Proposal cover (final to client)

When the draft is approved and Maaz is sending the actual proposal to the client (direct OR via AIMS):

```
<Customer Name>: <Proposal Topic> - Proposal SRE-<YYYY><NNN>
```

Examples (synthesized from project-numbering pattern observed):
- `Q-Chem: SGRU/SRU 2028 Feed Study and SRU/AGRU Training - Proposal SRE-2026121`
- `Phillips 66 Rodeo: GC Testing - Proposal SRE-2026122`

The `SRE-YYYY<NNN>` format ties to SRE's project numbering (matches the M365 group naming pattern from the audit).

---

## Pattern 7 — Internal urgency tags

When Maaz needs to flag an internal email as needing fast turnaround:

```
[URGENT] <subject>
```

Used sparingly. Real example from audit:
- `Re: [URGENT] Approval to purchase flight and hotel - SAMSUNG Adnoc Ruwais - 2024192`

Don't overuse — once `[URGENT]` becomes routine, it stops working.

---

## Pattern 8 — External tags (RFQ, NDA, contract refs)

For tender / RFQ correspondence:

```
RFQ <RFQ-Number>: <Customer> - <Topic>
```

```
NDA v<N>: <Counterparty> - <Topic>
```

Examples (from audit):
- `Re: KOC RFP-2103367: Request for Expression of Interest as Licensor/Technology Provider for KOC Project.`
- `Re: TGTU - on BOOT Basis` (technical inquiry without formal RFQ number)

The `RE:` is auto-prepended when replying — don't add it yourself in a new thread.

---

## Pattern 9 — Health check / silence-breaker

For re-engaging silent customers (paired with R5/R22 monitor output):

```
<Customer> - checking in
```

Or more specific:

```
<Customer> - <last-topic> status?
```

Examples (hypothetical, fitting the pattern):
- `KOC - status update on RFP-2103367`
- `Tidewater - touching base after February`
- `CEPSA SRU Testing Tender - any updates?`

## Decision rules

| Situation | Subject line |
|---|---|
| Drafting an email TO AIMS for them to review before client send | `DRAFT for your approval - <Customer> (<Contact>): <Topic>` |
| Sending final approved proposal to client direct | `<Customer>: <Topic> - Proposal SRE-<YYYY><NNN>` |
| Sending final approved proposal via AIMS to client | Same as above; AIMS contact in To:, client in CC (per operator preference) |
| Forwarding an external email to internal team | `FW: <original subject>` |
| AR escalation (operator wants to chase) | `<Customer> AR - Reminder #<N>` or reply to existing thread with `Re:` prefix |
| Year-end audit response | Reply to incoming with `Re:` — the auditor sets the subject |
| Post-meeting recap | `<Meeting topic> - notes and action items` |
| Health check | `<Customer> - checking in` OR `<Customer> - <last topic> status?` |
| NDA chaser | `NDA v<N>: <Counterparty> - status` |
| Internal coord | Direct, no prefix: e.g. "Updated Q-Chem timeline" |

---

## Anti-patterns

- ❌ All-lowercase subjects ("re: q-chem update") — looks unprofessional
- ❌ ALL-CAPS subjects (except specific tags like `[URGENT]`) — reads as shouting
- ❌ Multiple "RE: RE: FW: RE:" in front of subjects — clean up when responding
- ❌ Vague subjects ("Quick question", "Following up") — recipient can't triage
- ❌ Project-code-only subjects ("2024119") — needs customer name AND topic
- ❌ Trailing whitespace (audit found this pattern in calendar — also affects email subjects)
- ❌ Mojibake (UTF-8 dash → `â€"`) — clean encoding before sending
